The in-repo TypeScript part integrates that model into fps/ as a `usePunchClassifier` React hook that consumes smoothed `PoseKeypoint[]` from `useOneEuroFilter` and returns `{ type, confidence, speed }` at 30fps.

**Critical findings from the existing codebase:**
- `onnxruntime-web` is NOT installed in fps/ (confirmed: `npm ls onnxruntime-web` returns empty). It must be added.
- `fps/src/lib/velocity.ts` already exists (copied in Phase 13). `LANDMARK` constants are already defined there for indices 11-16, 23-24.
- `fps/src/App.tsx` is fully wired for Phase 13 (usePose + useOneEuroFilter + CalibrationScreen). Phase 13.1 adds one new hook call and one new outbound message type.
- `shared/protocol.ts` does not yet define a `MsgPunchDetected` or equivalent punch event type. This must be added as part of Phase 13.1's protocol work.
- No `ml/` directory exists. It must be created.

**BoxingVI dataset critical finding:** BoxingVI uses AlphaPose COCO-17 format (17 joints, indices 0–16) whereas MediaPipe uses 33 landmarks with a DIFFERENT numbering. Direct index use from BoxingVI data requires a remapping step before any cross-dataset training can proceed. [CITED: BoxingVI paper arXiv:2511.16524, AlphaPose COCO output.md]

**Primary recommendation:** Use a 5-layer temporal MLP (not 1D CNN) as the primary architecture for the <500KB constraint. 1D CNN is the fallback if accuracy is insufficient. Train on re-extracted MediaPipe keypoints from BoxingVI videos + webcam recordings, with AlphaPose data as an optional supplement after remapping.

## A. Data Pipeline Findings

### A1: BoxingVI Dataset Format

**Finding:** BoxingVI uses AlphaPose COCO-17 keypoint format. 17 joints (indices 0–16). The dataset normalizes by dividing x/y by video width/height — NOT by shoulder-width. Sequences are padded to 25 frames (the max frames for a punch at 30fps). Classes: Cross, Jab, Lead Hook, Lead Uppercut, Rear Hook, Rear Uppercut. [CITED: arXiv:2511.16524]

**AlphaPose COCO-17 index mapping:**

| COCO Index | Joint | MediaPipe Index |
|-----------|-------|-----------------|
| 0 | Nose | 0 |
| 5 | Left Shoulder | 11 |
| 6 | Right Shoulder | 12 |
| 7 | Left Elbow | 13 |
| 8 | Right Elbow | 14 |
| 9 | Left Wrist | 15 |
| 10 | Right Wrist | 16 |
| 11 | Left Hip | 23 |
| 12 | Right Hip | 24 |

[CITED: AlphaPose docs/output.md, COCO dataset spec]

**Critical consequence:** BoxingVI keypoints are 2D (x, y, confidence) normalized to [0,1] by image dimensions. MediaPipe world landmarks are 3D metric (x, y, z in meters, hip-center origin). They cannot be used together without re-extraction. The recommended approach:

**Option A (recommended):** Re-run MediaPipe PoseLandmarker on the BoxingVI raw videos to extract 3D world landmarks in MediaPipe format. Discard the AlphaPose annotations entirely. This ensures training and inference use identical landmark semantics.

**Option B:** Use AlphaPose COCO-17 data after extracting the 9 joints that map to MediaPipe indices, discarding z (since BoxingVI is 2D). Train on (x, y) only, add z during inference as a derived feature. Higher risk of domain mismatch.

**Option A is strongly preferred.** BoxingVI videos are from YouTube sparring sessions — the same domain as webcam boxing. [ASSUMED]

### A3: Normalization Formula

The standard approach for skeleton-based action recognition is to normalize relative to the body frame to achieve person-independence: [CITED: Multiple skeleton action recognition papers; ASSUMED for exact formula]

```python
def normalize_keypoints(kps: np.ndarray) -> np.ndarray:
    """
    kps: shape (T, J, 3) — T frames, J joints, xyz in meters
    Returns: shape (T, J, 3) normalized to shoulder-midpoint origin, scaled by shoulder width
    """
    # Shoulder midpoint = mean of left_shoulder (idx 0 in extracted subset) and right_shoulder (idx 1)
    # In the extracted 8-joint subset: shoulder order depends on extraction code
    left_shoulder = kps[:, 0, :]   # index into extracted subset, not MediaPipe global index
    right_shoulder = kps[:, 1, :]

    midpoint = (left_shoulder + right_shoulder) / 2.0  # shape (T, 3)
    shoulder_width = np.linalg.norm(right_shoulder - left_shoulder, axis=1, keepdims=True)  # (T, 1)
    shoulder_width = np.clip(shoulder_width, 1e-6, None)  # prevent division by zero

    # Translate: subtract midpoint from all joints
    normalized = kps - midpoint[:, np.newaxis, :]   # broadcast (T, 1, 3)

    # Scale: divide by shoulder width
    normalized = normalized / shoulder_width[:, np.newaxis, :]

    return normalized
```

**This same normalization MUST be applied identically in `usePunchClassifier.ts` at inference time.** The hook must compute the shoulder midpoint and shoulder width from the current frame's smoothed keypoints and normalize the sliding window before calling `session.run()`. [ASSUMED — exact formula confirmed as standard; project-specific ordering needs verification during training]

### A4: Horizontal-Flip Augmentation

For left/right symmetry augmentation during training:

```python
def flip_sequence(kps: np.ndarray, label: str) -> tuple[np.ndarray, str]:
    """
    kps: (T, J, 3) in the extracted subset order:
         [left_shoulder, right_shoulder, left_elbow, right_elbow, left_wrist, right_wrist, left_hip, right_hip]
    """
    flipped = kps.copy()
    # Swap paired joints: (left_shoulder↔right_shoulder), (left_elbow↔right_elbow), etc.
    SWAP_PAIRS = [(0, 1), (2, 3), (4, 5), (6, 7)]
    for i, j in SWAP_PAIRS:
        flipped[:, i, :], flipped[:, j, :] = kps[:, j, :].copy(), kps[:, i, :].copy()
    # Negate x coordinate (mirror)
    flipped[:, :, 0] *= -1

    # Swap punch label: jab↔cross, hook_l↔hook_r
    LABEL_SWAP = {'jab': 'cross', 'cross': 'jab', 'hook_l': 'hook_r', 'hook_r': 'hook_l', 'guard': 'guard'}
    return flipped, LABEL_SWAP[label]
```

[ASSUMED — standard skeleton augmentation pattern]

### A5: Sliding Window Size

**BoxingVI finding:** Maximum punch duration at 30fps is 25 frames (~833ms). [CITED: arXiv:2511.16524]

**Recommended window:** T=20 frames at 30fps = 667ms. This covers most punches while keeping the input tensor small.

- Stride during training: 5 frames (50% overlap) for data augmentation
- Stride during inference: 1 frame (classify every new frame, return the most recent prediction)

**Input tensor shape at inference:** `(1, 20, 8, 3)` = batch × frames × joints × coords. Flattened for the MLP: `(1, 480)`.

## B. Model Architecture Findings

### B1: Temporal MLP Architecture (Primary Recommendation)

For 480-dimensional input (20 frames × 8 joints × 3 coords), classifying 5 classes:

```python
import torch.nn as nn

class PunchMLP(nn.Module):
    def __init__(self, input_dim=480, num_classes=5):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(input_dim, 256),
            nn.ReLU(),
            nn.Dropout(0.3),
            nn.Linear(256, 128),
            nn.ReLU(),
            nn.Dropout(0.3),
            nn.Linear(128, 64),
            nn.ReLU(),
            nn.Linear(64, num_classes),
        )

    def forward(self, x):
        # x: (batch, T, J, C) -> flatten to (batch, T*J*C)
        return self.net(x.view(x.size(0), -1))
```

**Size estimate:** Linear layers: (480×256 + 256) + (256×128 + 128) + (128×64 + 64) + (64×5 + 5) weights = ~155K parameters × 4 bytes (FP32) ≈ 620KB FP32. After INT8 quantization (4× reduction): **~155KB**. Well within the 500KB limit. [ASSUMED — calculated from layer sizes; actual ONNX overhead adds ~5-10KB]

**Why MLP over CNN for this phase:** MLP is simpler to export, has no conv ops that vary by opset, and at 155KB quantized is far under budget. CNN is needed only if MLP accuracy is below 80% after tuning. [ASSUMED]

### B2: 1D CNN Alternative Architecture (Fallback)

If MLP accuracy is insufficient (<80%) for capturing temporal dynamics:

```python
class PunchCNN1D(nn.Module):
    def __init__(self, joint_dim=24, T=20, num_classes=5):
        # joint_dim = 8 joints × 3 coords = 24 channels
        super().__init__()
        self.conv = nn.Sequential(
            nn.Conv1d(joint_dim, 64, kernel_size=3, padding=1),
            nn.ReLU(),
            nn.Conv1d(64, 128, kernel_size=3, padding=1),
            nn.ReLU(),
            nn.AdaptiveAvgPool1d(1),   # global average over time
        )
        self.classifier = nn.Sequential(
            nn.Linear(128, 64),
            nn.ReLU(),
            nn.Linear(64, num_classes),
        )

    def forward(self, x):
        # x: (batch, T, J, C) -> reshape to (batch, J*C, T) for Conv1d
        b, T, J, C = x.shape
        x = x.reshape(b, J * C, T)
        x = self.conv(x)
        x = x.squeeze(-1)
        return self.classifier(x)
```

**Size estimate:** Conv1d params: (24×64×3 + 64) + (64×128×3 + 128) = ~29K weights + classifier ~9K = ~38K params × 4 bytes ≈ 152KB FP32, ~38KB quantized. Smaller than MLP but more complex to tune. [ASSUMED — calculated from architecture]

### B5: INT8 Quantization

Use `onnxruntime.quantization` (not `torch.quantization`) — it operates on the ONNX graph directly and is the tool onnxruntime-web consumes. [CITED: onnxruntime.ai/docs/performance/model-optimizations/quantization.html]

```python
from onnxruntime.quantization import quantize_dynamic, QuantType

quantize_dynamic(
    model_input="punch_classifier.onnx",
    model_output="punch_classifier_int8.onnx",
    weight_type=QuantType.QInt8,
)
```

**Expected result:** FP32 MLP at ~620KB → INT8 at ~155KB. FP32 CNN at ~152KB → INT8 at ~38KB. Both well under the 500KB target. [ASSUMED — standard 4× compression ratio for INT8; actual depends on model graph]

**Warning:** Dynamic quantization quantizes weights only (not activations). For a tiny MLP like this, this is preferred over static quantization (which requires a calibration dataset and adds complexity). [CITED: onnxruntime quantization docs]

**Approach B (self-hosted WASM, more robust):** Copy WASM files from node_modules to fps/public/ and use vite-plugin-wasm or viteStaticCopy. Adds build complexity.

For Phase 13.1, **Approach A is simpler** and avoids Vite build config changes. The CDN WASM files are versioned and stable. [ASSUMED — CDN approach is acceptable for this project's game context; if offline/LAN play is required, switch to Approach B]

### C4: InferenceSession Lifecycle in a React Hook

Session creation is async and expensive (WASM initialization). It must happen once at mount, not on every render:

```typescript
import { useEffect, useRef, useState } from 'react';
import * as ort from 'onnxruntime-web';

// Set wasmPaths before any session creation — do this at module level or in a one-time setup
ort.env.wasm.wasmPaths = 'https://cdn.jsdelivr.net/npm/onnxruntime-web@1.26.0/dist/';

export function usePunchClassifier(keypoints: PoseKeypoint[] | null): PunchClassifierResult {
  const sessionRef = useRef<ort.InferenceSession | null>(null);
  const bufferRef = useRef<PoseKeypoint[][]>([]);  // ring buffer, max T=20 frames
  const [result, setResult] = useState<PunchClassifierResult>({ type: null, confidence: 0, speed: 0 });

  // Load model once on mount
  useEffect(() => {
    let cancelled = false;
    async function loadModel() {
      try {
        const session = await ort.InferenceSession.create('/models/punch_classifier_int8.onnx', {
          executionProviders: ['wasm'],
        });
        if (!cancelled) sessionRef.current = session;
      } catch (err) {
        console.error('[usePunchClassifier] model load failed:', err);
      }
    }
    loadModel();
    return () => { cancelled = true; };
  }, []);

  // Run inference on each new keypoint frame
  useEffect(() => {
    if (!keypoints || !sessionRef.current) return;

    // Update ring buffer
    bufferRef.current = [...bufferRef.current, keypoints].slice(-20);  // keep last T=20
    if (bufferRef.current.length < 20) return;  // wait for full window

    // Build input tensor
    const JOINT_INDICES = [11, 12, 13, 14, 15, 16, 23, 24];
    const T = 20, J = 8, C = 3;
    const data = new Float32Array(T * J * C);
    for (let t = 0; t < T; t++) {
      for (let j = 0; j < J; j++) {
        const kp = bufferRef.current[t][JOINT_INDICES[j]];
        data[t * J * C + j * C + 0] = kp.x;
        data[t * J * C + j * C + 1] = kp.y;
        data[t * J * C + j * C + 2] = kp.z;
      }
    }
    // NOTE: normalize data here (translate to shoulder midpoint, scale by shoulder width)
    // ... normalization code ...

    const tensor = new ort.Tensor('float32', data, [1, T, J, C]);
    sessionRef.current.run({ input: tensor }).then((output) => {
      const logits = output.logits.data as Float32Array;
      const probs = softmax(logits);
      const maxIdx = probs.indexOf(Math.max(...Array.from(probs)));
      const confidence = probs[maxIdx];
      const LABELS: PunchType[] = ['jab', 'cross', 'hook_l', 'hook_r', 'guard'];
      setResult({ type: confidence > 0.7 ? LABELS[maxIdx] : null, confidence, speed: 0 });
    }).catch(() => {});
  }, [keypoints]);

  return result;
}
```

### C7: vs/cross Disambiguation Using z-coordinate

MediaPipe world landmarks provide z in meters (depth from camera). For jab/cross disambiguation:
- Jab: lead hand extends forward — z decreases (hand moves toward camera)
- Cross: rear hand extends forward from a rotated torso position

The z-coordinate of the extending wrist relative to the shoulder plane is the key signal. The model should learn this from training data (z is included in the 3-channel input). No hand-coded z rule is needed if training data is diverse. [ASSUMED]

### Pattern: usePunchClassifier ring buffer

The buffer must use `useRef` to avoid React render loops. The inference result uses `useState` so components re-render on new detections:

```typescript
// Source: onnxruntime-web official example + adaptation
const bufferRef = useRef<PoseKeypoint[][]>([]);

// In useEffect([keypoints]):
bufferRef.current.push(keypoints);
if (bufferRef.current.length > WINDOW_SIZE) {
  bufferRef.current.shift();  // O(n) but n=20, negligible
}
```

### Anti-Patterns to Avoid

- **Creating InferenceSession on every render:** Session creation is ~100-500ms. Store in `useRef`, create once in `useEffect([], [])`.
- **Sharing one ONNX session across multiple hook instances:** Each `usePunchClassifier` creates its own session. If multiple instances were needed (not the case here), they'd need a shared context.
- **Running session.run() synchronously:** `session.run()` returns a Promise. Use `.then()` or `async/await` inside a `useEffect` — never await in the render function.
- **Applying normalization in training but not in inference (or vice versa):** The normalization formula (shoulder midpoint + shoulder width scaling) must be bit-for-bit identical in both the Python training pipeline and the TypeScript hook.
- **Using image-space landmarks for the buffer:** `usePose` returns both `keypoints` (worldLandmarks, 3D metric) and `imageKeypoints` (2D normalized). The classifier must consume `worldLandmarks` (the same `keypoints` that `useOneEuroFilter` smooths). [VERIFIED: fps/src/App.tsx line 32 — `useOneEuroFilter(pose.keypoints)` uses worldLandmarks]
- **Committing raw video data or FP32 .onnx to the repo:** BoxingVI videos are ~GB. FP32 .onnx is ~620KB. Only commit the quantized INT8 model (~155KB).

## Common Pitfalls

### Pitfall 1: Training/inference normalization mismatch
**What goes wrong:** Model trained on normalized data but inference passes raw metric coordinates. All 5 classes predict at equal ~20% probability regardless of pose.
**Why it happens:** Forgetting to apply the same shoulder-midpoint normalization at inference time.
**How to avoid:** Define `normalizeWindow(buffer, jointIndices)` as a shared function and use it in BOTH `train.py` and `usePunchClassifier.ts`. Write a test that runs the normalization on a known input and checks the output.
**Warning signs:** Confidence always ~0.2 (uniform distribution); no class ever reaches 0.7 threshold.

### Pitfall 2: InferenceSession not ready when first keypoints arrive
**What goes wrong:** First 100-500ms of frames are dropped because the session hasn't loaded yet. More importantly, if `keypoints` changes before the load completes, the effect cleanup runs and cancels — the session is never created.
**Why it happens:** Model load is async; first keypoint frames arrive immediately.
**How to avoid:** The `useEffect([], [])` (empty deps) loads the session independently. The `useEffect([keypoints])` checks `sessionRef.current` before running inference. If null, silently skip.
**Warning signs:** Classifier returns null indefinitely even after model should have loaded.

### Pitfall 3: onnxruntime-web WASM files not found (404)
**What goes wrong:** `InferenceSession.create` throws "failed to load wasm" or similar. The WASM binary files are not being served.
**Why it happens:** By default, onnxruntime-web expects WASM files adjacent to its JS bundle. In a Vite project, they end up in the wrong directory.
**How to avoid:** Set `ort.env.wasm.wasmPaths` to the CDN URL before any session creation. Do this at module initialization level (top of usePunchClassifier.ts or in a shared ort-config.ts).
**Warning signs:** Console error containing "ort-wasm" and "404" or "failed to fetch".

### Pitfall 4: Buffer warming period causes false positives at match start
**What goes wrong:** First 19 frames (before buffer fills) are skipped correctly (type=null). But the first full-window inference may classify a random pose as a punch.
**Why it happens:** The buffer fills with calibration/neutral poses, not guard poses. The model sees an ambiguous transition.
**How to avoid:** After CalibrationScreen exits and 'match' phase starts, reset the buffer: `bufferRef.current = []`. The hook naturally suppresses output for the first 20 frames (buffer warm-up period).
**Warning signs:** Punch events fire immediately at match start before player has moved.

### Pitfall 5: AlphaPose COCO indices used directly as MediaPipe indices
**What goes wrong:** Training pipeline extracts joint 5 (COCO left shoulder) but at inference time the hook reads index 11 (MediaPipe left shoulder). The model learns from one joint, inference reads a completely different body part.
**Why it happens:** The two skeleton formats use different numbering. AlphaPose COCO shoulder=5; MediaPipe shoulder=11.
**How to avoid:** Either re-extract all data using MediaPipe (Option A, recommended), OR carefully define the joint extraction mapping once in a constants file used by BOTH the Python pipeline and the TypeScript hook.
**Warning signs:** Model trains to >90% accuracy but achieves ~20% in the browser — data/inference mismatch.

### Pitfall 6: shared/protocol.ts overwritten by cargo test regeneration
**What goes wrong:** `MsgPunchDetected` added to `shared/protocol.ts` is erased the next time `cargo test` regenerates the TypeScript types from Rust.
**Why it happens:** The file header says "do not edit by hand — generated by ts-rs".
**How to avoid:** Do NOT add punch types to `shared/protocol.ts`. Define `PunchType` and `MsgPunchDetected` locally in `fps/src/hooks/usePunchClassifier.ts`. [VERIFIED: shared/protocol.ts line 1 — "Generated by ts-rs from engine/engine-core/src/protocol.rs"]

### Normalization at inference (TypeScript)
```typescript
// Normalize the 20-frame window before building the tensor
function normalizeWindow(buffer: PoseKeypoint[][], jointIndices: number[]): Float32Array {
  const T = buffer.length, J = jointIndices.length, C = 3;
  const data = new Float32Array(T * J * C);

  for (let t = 0; t < T; t++) {
    const frame = buffer[t];
    const lSh = frame[jointIndices[0]];  // left shoulder
    const rSh = frame[jointIndices[1]];  // right shoulder

    const mx = (lSh.x + rSh.x) / 2, my = (lSh.y + rSh.y) / 2, mz = (lSh.z + rSh.z) / 2;
    const sw = Math.sqrt((rSh.x - lSh.x) ** 2 + (rSh.y - lSh.y) ** 2 + (rSh.z - lSh.z) ** 2);
    const scale = sw > 1e-6 ? sw : 1;

    for (let j = 0; j < J; j++) {
      const kp = frame[jointIndices[j]];
      const base = t * J * C + j * C;
      data[base + 0] = (kp.x - mx) / scale;
      data[base + 1] = (kp.y - my) / scale;
      data[base + 2] = (kp.z - mz) / scale;
    }
  }
  return data;
}
```

## Key Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D1 | Use temporal MLP (not 1D CNN) as primary architecture | Simpler to export; ~155KB quantized (well under 500KB); sufficient for 480-dim input |
| D2 | Re-extract keypoints from BoxingVI using MediaPipe PoseLandmarker (not AlphaPose annotations) | AlphaPose COCO-17 and MediaPipe-33 use different joint indices and coordinate systems; re-extraction ensures train/inference parity |
| D3 | Sliding window T=20 frames at 30fps (667ms) | BoxingVI max punch duration is 25 frames; T=20 covers most punches while keeping tensor small |
| D4 | Normalization: translate to shoulder midpoint, scale by shoulder width | Person-independent; works across diverse users; must match exactly between training and inference |
| D5 | onnxruntime-web WASM CDN approach for Phase 13.1 (not self-hosted WASM in public/) | Avoids Vite build config complexity; model is small; CDN WASM is versioned and stable |
| D6 | Commit quantized ONNX to fps/public/models/ (~155KB) | Small enough for git; simpler than HuggingFace Hub; no LFS needed |
| D7 | Define PunchType and MsgPunchDetected locally in fps/ (not in shared/protocol.ts) | shared/protocol.ts is auto-generated from Rust; adding TS types there would be overwritten by next cargo test run |
| D8 | speed field in usePunchClassifier output computed via computeWristVelocity, not model output | Existing utility already handles timing; model doesn't need to output continuous values |
| D9 | Confidence threshold 0.7 for punch detection (below = type: null) | Starting value; needs empirical tuning after training |

## Open Questions (RESOLVED)

1. **Is BoxingVI raw video data accessible for re-extraction?**
   RESOLVED: BoxingVI raw videos are publicly available on Google Drive (no request needed): https://drive.google.com/drive/folders/1Vyl8twJQ1qkqEPwhvfsrJsJ8nLQ92uoy. Re-extraction using MediaPipe Python is required (AlphaPose annotations are incompatible). `extract_keypoints.py` handles this. Webcam-only recordings remain a valid fallback if the Drive link expires.

2. **Should guard be a 5th class or handled by absence of punch?**
   RESOLVED: Guard is a positive 5th class. Plans treat it as a distinct label in `PUNCH_CLASSES = ['jab', 'cross', 'hook_l', 'hook_r', 'guard']`. This enables explicit guard feedback in the game UI and is architecturally cleaner than "not-a-punch" suppression logic.

3. **Does onnxruntime-web 1.26.0 support opset 17?**
   RESOLVED: Yes — onnxruntime-web 1.26.0 supports ONNX opset 17 (onnxruntime >= 1.14 added opset 17 support). Plans use `opset_version=17` in `torch.onnx.export`. Fallback to opset 12 documented in ml/README.md.

4. **Where should WASM files be served in production (Docker)?**
   RESOLVED: Phase 13.1 uses jsdelivr CDN via `ort.env.wasm.wasmPaths` (development + initial deployment). The Dockerfile WASM self-hosting step is explicitly deferred to Phase 14 polish. Plans set `wasmPaths` to CDN only.
