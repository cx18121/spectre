<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MsgFpsHit → trauma/snap-back trigger | Server controls damage value and punch_type; attacker controls shake magnitude |
| AudioContext creation | Created on first play call — must be triggered by user gesture, not server message |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-03-01 | Denial of Service | Camera shake — server sends high-damage MsgFpsHit rapidly | mitigate | `trauma = Math.min(1.0, trauma + amount)` — trauma is capped at 1.0 regardless of damage value. Quadratic shake (trauma²) limits maximum displacement to MAX_TRANSLATE_SHAKE = 0.05 meters. |
| T-14-03-02 | Denial of Service | Audio — rapid MsgFpsHit events create unbounded AudioBufferSourceNode objects | mitigate | Each play*() creates nodes, connects, starts, stops, then GC collects (fire-and-forget). AudioContext has a built-in limit on concurrent nodes (~64 in Chrome). Rapid hits will drop audio — acceptable for game UX; not a crash risk. |
| T-14-03-03 | Tampering | punch_type string from server drives audio selection | accept | Only three branches: "blocked" → playBlocked(), any other → playImpact(). Server sending unexpected punch_type falls through to playImpact() — no code path executes untrusted strings. |
| T-14-03-04 | Elevation of Privilege | AudioContext autoplay bypass | accept | AudioContext is created lazily inside playImpact/playBlocked, which are triggered by game events. The first game event occurs after the player has interacted with the page (they started calibration), satisfying the browser's user-gesture requirement. |
</threat_model>
